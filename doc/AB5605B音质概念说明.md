## 源码分析

```c
#define BT_CODEC (CODEC_SBC)                          // SBC：强制开启
                | (CODEC_AAC * BT_A2DP_AAC_AUDIO_EN)  // AAC：条件开启 × 0或1
                | (CODEC_MSBC * BT_HFP_MSBC_EN)       // mSBC：条件开启 × 0或1
                | (CODEC_PLC * BT_PLC_EN)             // PLC：条件开启 × 0或1
```

### 1. 各编码的类型与速率

| 编码     | 宏值   | 用途                                       | 典型码率                              | 延迟       | 频宽       |
| -------- | ------ | ------------------------------------------ | ------------------------------------- | ---------- | ---------- |
| **SBC**  | `0x01` | A2DP 音乐（强制支持）                      | **328 kbps**（标准）<br>最大 510 kbps | ~150-200ms | 20Hz-20KHz |
| **AAC**  | `0x02` | A2DP 音乐（可选）                          | **256 kbps**                          | ~90-120ms  | 20Hz-20KHz |
| **mSBC** | `0x04` | HFP 通话（宽带语音）                       | **64 kbps**                           | ~30ms      | 50Hz-8KHz  |
| **PLC**  | `0x08` | 丢包补偿（非独立编码，配合 SBC/mSBC 使用） | 无独立码率                            | -          | -          |

- **SBC 是蓝牙规范强制要求**，所以 `CODEC_SBC` 不带条件，始终存在
- **AAC 和 mSBC 是互斥用途**：AAC 走 A2DP（音乐），mSBC 走 HFP（通话），不冲突
- **PLC 是辅助功能**：不是独立编码，是增强 SBC/mSBC 抗丢包的算法
- 如果你把 `BT_A2DP_AAC_AUDIO_EN` 改为 1，AAC 就会加入编解码器协商列表，蓝牙连接时手机会选择两端都支持的编码

### 2. 音乐走 A2DP → SBC vs AAC

SBC 和 AAC **走的是同一条路（A2DP 音乐通道）**，**二选一**：

```
手机 → A2DP连接 → 编解码器协商 → 选SBC或AAC → 解码 → DAC输出
                        ↑
             两端都支持的编码列表
```

**工作流程**：
1. 音箱告诉手机：我支持 `{SBC, AAC}`
2. 手机从交集里选一个它认为最好的（通常 iOS 优先 AAC，Android 倾向 SBC）
3. **只选中一个**，另一个不用

**结果**：开启 AAC 后，SBC 不是被"替换"，而是作为**备选**——手机不支持 AAC 时就 fallback 到 SBC。

---

#### 2.1 Q：为什么 AAC还会比  SBC效果要好

为什么 AAC 效率更高

| 维度             | SBC                                              | AAC                                                    |
| ---------------- | ------------------------------------------------ | ------------------------------------------------------ |
| **算法年代**     | 1998年，蓝牙SIG强制的最低标准                    | 1997年，MPEG-2/4 标准，持续迭代优化                    |
| **编码原理**     | 简单的子带编码（Sub-band Coding），4/8个等宽子带 | 改进的MDCT + 心理声学模型                              |
| **心理声学模型** | **无**，所有频段统一量化                         | **有**，利用人耳掩蔽效应，把听不到的噪音藏在信号里     |
| **处理方式**     | 每个子带平均分配bit                              | 按听觉重要性分配bit，人声/中频给更多，高频掩蔽部分少给 |

实际听感差异

```
人耳最敏感范围：2KHz - 5KHz（人声、乐器泛音）

SBC：各频段均匀失真，2-5KHz 也可能有量化噪声 → 听感"粗糙、数码味"
AAC：2-5KHz 优先保证精度，把量化噪声推到人耳不敏感的高频 → 听感"干净、通透"
```

总结： **码率高≠音质好。AAC 用少了 22% 的码率，因为算法更聪明（心理声学模型+MDCT），反而比 SBC 保留了更多有用的音频信息。**

### 当前AB5605BVS蓝牙5.0无损

| 对比维度           | 当前 AB5605B         | 「无损」要求                                    | 差距                  |
| ------------------ | -------------------- | ----------------------------------------------- | --------------------- |
| **SoC 架构**       | 单核 RISC-V @ 120MHz | 多核（DSP + MCU），如高通 QCC5171 @ 双核 320MHz | ❌ 算力差 5-10 倍      |
| **蓝牙版本**       | BR/EDR +BLE 5.3      | （需支持 LE Audio / LC3）                       | ❌ 差 2 个大版本       |
| **音乐编解码**     | SBC only（328kbps）  | LDAC（990kbps）或 aptX HD（576kbps）或 LHDC     | ❌ 码率差 2~3 倍       |
| **通话编解码**     | CVSD（64kbps, 4KHz） | mSBC（64kbps, 8KHz）+ AEC                       | ⚠️ mSBC 代码有，但关着 |
| **LE Audio (LC3)** | ❌ 不支持             | ✅ 蓝牙 5.2 核心卖点                             | ❌ 完全缺失            |
| **DAC 采样率**     | 48KHz                | 96KHz / 192KHz                                  | ❌ 差 2~4 倍           |
| **DAC 位深**       | 16bit                | 24bit / 32bit                                   | ❌ 差 1 倍             |
| **DAC 信噪比**     | ~90dB（内置）        | ~120dB（外置 DAC 芯片）                         | ❌ 差 30dB             |
| **音频处理能力**   | 无 DSP，CPU 自算     | 独立 DSP 核，硬件加速 EQ/DRC/分频               | ❌ 无硬件加速          |
| **TWS 真无线**     | ❌ 关着，有代码但受限 | ✅ 成熟方案（主从转发）                          | ⚠️ 有代码可试          |
| **一拖二**         | ❌ `BT_2ACL_EN 0`     | ✅ 多设备切换                                    | ⚠️ 可开启但功能弱      |
| **编解码器授权**   | 无授权费用           | 高通/SONY/华为每芯片 $0.5-2                     | ❌ 法律门槛            |

核心差距

| 编码        | 典型码率            | 当前SDK    | 无损产品                  |
| ----------- | ------------------- | ---------- | ------------------------- |
| **SBC**     | 328 kbps（最大510） | ✅ 强制支持 | ✅ 强制支持                |
| **AAC**     | 256 kbps            | ❌ 关闭     | ✅ 基本标配                |
| **aptX**    | 384 kbps            | ❌ 不支持   | ✅ 需高通芯片              |
| **aptX HD** | **576 kbps**        | ❌ 不支持   | ✅ 需高通芯片              |
| **LDAC**    | **990 kbps**        | ❌ 不支持   | ✅ 需**索尼认证**+专用芯片 |
| **LHDC**    | **900 kbps**        | ❌ 不支持   | ✅ 需华为认证              |
| **LC3**     | 160-320 kbps        | ❌ 不支持   | ✅ 需 LE Audio 芯片        |

**缺口在哪**：

```
真正"无损"需要的码率: CD = 1411 kbps (16bit/44.1K)
                         LDAC 990kbps ≈ "Hi-Res" (约70%)
                         aptX HD 576kbps ≈ "高清"（约41%）
                         ────────
当前SDK的能力:           SBC 328kbps ≈ 基础音质（约23%）
                         差了 3 ~ 4 倍的码率
```

**编解码器是硬件绑定的**：

| 编码           | 拥有者            | 需要什么                                |
| -------------- | ----------------- | --------------------------------------- |
| aptX / aptX HD | **高通 Qualcomm** | 高通蓝牙芯片（如 QCC 系列）+ **授权费** |
| LDAC           | **索尼 Sony**     | 索尼认证芯片 + **授权费**               |
| LHDC           | **华为/盛微**     | 特定芯片 + 授权                         |
| LC3            | 蓝牙SIG（标准）   | 蓝牙5.2+ 控制器 + **新协议栈**          |

**AB5605B 是低端 RISC-V 芯片，没有这些编解码器的硬件加速器，闭源库也不包含它们。软件加不了，因为 SBC/AAC 以外都是专利闭源方案。**
