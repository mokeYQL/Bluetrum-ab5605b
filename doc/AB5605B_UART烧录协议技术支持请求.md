## 二、为什么现有OTA方案不可用

### 2.1 芯片架构限制（根因分析）

AB5605B 是**单分区、无独立Bootloader**架构：

```
AB5605B Flash 布局 (512KB)
┌──────────────────────┐ 0x10000000
│  init 启动向量  512B │
├──────────────────────┤ 0x10000200
│  代码区 492KB    ←── 唯一的程序存储区
│  (Main / A2DP /      │     升级时被直接覆写
│   UART ISR /         │
│   updateproc)        │
├──────────────────────┤ 0x1007B000
│  CM参数区 20KB       │
├──────────────────────┤ FLASH_SIZE
│  (死区)              │
├══════════════════════┤ 0x11000000
│  MP3资源区           │
└──────────────────────┘
```

| 特性 | AB5605B |
|------|---------|
| Flash分区 | 单Bank 512KB |
| Bootloader | ❌ 无独立Bootloader |
| 升级方式 | App自己擦写自己 |
| 断电保护 | ❌ 断电即变砖 |
| 变砖恢复 | ISP烧录器拆机 |

### 2.2 SDK内置FOTA（BT_AB_FOT_EN / LE_AB_FOT_EN）的风险

SDK内置的 `app_fota.c` 只是蓝牙传输协议层，不是安全升级机制：

```c
// app_fota.c — FOTA升级流程
AT(.text.fot.cache)
void app_fota_write(void *buf, u32 addr, u32 len) {
    fot_write(buf, addr, len);  // ← App在运行中直接擦写自身Flash代码区
}
```

**变砖时序**:
```
升级进行中（~4秒窗口）
  │
  ├── ① 擦除 0x10000200 (启动代码)    ← 最早被破坏
  ├── ② 擦除 0x10001600 (.comm副本)   ← UART ISR 源数据被毁
  ├── ③ 擦除 0x10004400 (主代码区)    ← main/事件处理被毁
  │
  └── ⚡ 此时断电
      │
      ├── 重新上电 → init区可能已被擦 → 无法启动
      ├── .comm Flash副本已被覆盖 → RAM中的UART ISR加载垃圾数据 → 串口不通
      ├── main()/func_update()已残缺 → 无法进入升级流程
      └── 结论: 💀 变砖，只能拆机用ISP烧录器恢复
```

### 2.3 风险评估结论

| 升级通道 | 断连风险 | 变砖风险 | 可否量产使用 |
|----------|---------|---------|-------------|
| SPP/BLE FOTA (无线) | 高 | 高 | ❌ 不可接受 |
| USB U盘升级 | 中 (断电) | 中 | ❌ ESP32S3 PSRAM占用2.5M |
| UART串口升级 (有线) | 极低 | 仍存在(断电) | ⚠️ 需原厂支持 |

## 三、技术诉求

### 3.1 核心诉求

> **希望贵司提供 AB5605B 的 UART 下载/烧录上位机协议，以便我司 ESP32S3 可以通过串口（PB3）对 AB5605B 进行完整的固件烧录。**

### 3.2 需要的协议信息

参照蓝讯 Downloader v3.3.3 工具的通信协议，需要如下技术细节：

| 序号 | 需求内容 | 说明 |
|------|---------|------|
| 1 | 握手协议 | 进入下载模式的具体时序和命令序列 |
| 2 | 命令帧格式 | 完整的帧结构定义（含CRC算法） |
| 3 | Flash操作命令 | 擦除/写入/校验的具体命令编码 |
| 4 | 数据分包规则 | 每包大小、地址对齐要求 |
| 5 | CRC/校验算法 | 使用的校验算法和计算范围 |
| 6 | 错误处理机制 | 超时/重传/错误码定义 |
| 7 | 时序要求 | 各操作之间的等待时间要求 |

### 3.3 交付方式建议

|------|------|

## 四、量产解决方案

### 4.1 当前困局

```
量产流水线:
  PC端烧录器 → PB3(200R) → AB5605B  ← 出厂烧录 OK
                                        但售后升级?
                                           ↓
                                      没有可靠方案!
```

### 4.2 目标架构

```
┌─────────────────────────────────────────────────────┐
│                 ESP32S3 (联网主控)                   │
│                                                     │
│  ① WiFi/4G下载新固件                                 │
│  ② 存储到内部Flash/PSRAM                             │
│  ③ ★ 按原厂UART烧录协议打包发送 ← 此处需原厂支持    │
│  ④ CRC校验确认                                       │
│  ⑤ 指示AB5605B重启生效                               │
└─────────────────────┬───────────────────────────────┘
                      │ UART (PB3, 200R)
                      ▼
┌─────────────────────────────────────────────────────┐
│                 AB5605B (音频SoC)                    │
│                                                     │
│  内置UART下载协议 → 擦除Flash → 写入 → 校验 → 生效   │
│  (芯片BootROM实现，一直可用)                          │
└─────────────────────────────────────────────────────┘
```

### 4.3 安全保障

| 措施 | 说明 |
|------|------|
| 有线连接 | PB3直连，不依赖蓝牙，断连概率极低 |
| PB3+200R | 与官方Downloader相同的硬件连接方式，已验证稳定 |
| 双供电 | ESP32S3电池+外部电源双供电，升级期间不断电 |
| 可重试 | 升级失败后可重新握手重试 |
| CRC校验 | 每包校验，确保数据完整性 |
