# ESP32S3 串口升级 AB5605B 方案评估

## 现有条件

```
项目硬件上已有 ESP32S3（联网下载固件）
AB5605B 的 PA7 是 UART0 TX（目前用作 printf 日志输出）
```

## 三个可行方案对比

### 方案A：AB5605B 跑 UART_S_UPDATE 从机协议（可行但有风险）

SDK 内置了从机串口升级协议：

```c
// config.h 打开：
#define UART_S_UPDATE 1          // 从机模式
#define UART_UPD_BAUD 115200     // 波特率
#define UPDATE_UART_SEL UPDATE_UART0  // 用 UART0（PA7）
```

**工作流程：**

```
ESP32S3（主机）                    AB5605B（从机）
    │                                   │
    │  WiFi 下载固件                     │
    │                                   │
    ├── 握手 START_UPD ──────────────▶   │
    │                                   │
    ├── 命令: 检查升级模式 ──────────▶    │
    │◀── 响应: OK ──────────────────    │
    │                                   │
    ├── 命令: 发送数据(地址+长度+CRC) ─▶  │
    │   ├── 固件数据(512字节/包) ─────▶   │ → 写入自身 Flash
    │   └── ... (循环)                   │
    │                                   │
    ├── 命令: 读取状态 ──────────────▶    │
    │◀── 0xFF (完成) ────────────────    │
    │                                   │
    │   WDT_RST() ◀── 从机重启           │
```

**为什么比 BLE/SPP FOTA 更可靠？**

```
BLE/SPP FOTA：      无线连接 → 易断连 → 易变砖
UART 串口有线：     物理连接 → 不断连 → 大幅降低风险
```

**风险仍在：掉电**

```
升级中途拔 USB/断电 → Flash 写入中断 → 既有的程序被破坏 → 变砖
```

### 方案B：AB5605B 代码在 RAM 跑，ESP32S3 接管 Flash 擦写（更安全）

思路：AB5605B 启动时把升级程序加载到 RAM 里，然后释放 Flash 总线控制权，让 ESP32S3 直接写 Flash。

但 AB5605B **没有开放 Flash 总线给外部 MCU 访问的接口**（Flash 是内部 SPI，引脚不引出设备外部）。此方案**不可行**。

### 方案C：ESP32S3 直接操作 Flash 芯片（最安全，改硬件）

```
ESP32S3 GPIO ──▶ AB5605B Flash 芯片的 SPI 引脚
                      └─ SO/SI/SCK/CS 四线
```

ESP32S3 把自己模拟成 SPI Flash 编程器，不依赖 AB5605B 运行：

```
                    ┌──────────────┐
                    │  ESP32S3     │    WiFi 下载固件
                    │  (SPI 主机)   │◀─── 互联网
                    └──────┬───────┘
                           │ SPI 四线
                           ▼
                    ┌──────────────┐
                    │  SPI Flash   │    AB5605B 掉电/复位
                    │  (W25Qxx)    │    不影响烧录
                    └──────────────┘
                    AB5605B 处于复位状态
```

**优点：零变砖风险**，Flash 芯片独立编程，AB5605B 完全不参与。

**缺点：**
- 需要硬件连接 SPI Flash 引脚（可能需要飞线）
- ESP32S3 需要实现 SPI Flash 编程器逻辑
- Flash 封装不同（SOP8/SOP16）决定连接难度

## 推荐方案：方案A（UART_S_UPDATE）

```
开发阶段：    用 ISP 烧录器（USB 转 SPI）— 最稳妥
量产/升级时： ESP32S3 + UART 升级        — 串口有线可靠，变砖概率极低
             如果能保证不拔电，基本不会变砖
```

### 接线

```
ESP32S3 UART TX  ──▶ AB5605B PA7 (UART0 RX) 
ESP32S3 UART RX  ◀── AB5605B PA7 (UART0 TX)
                       注意：PA7 当前是 printf，需改为复用 UART 升级
                       或另选其他 UART 引脚（如 PA3/PA4 对应 UART1）
GND ────────────────── GND
```

### ESP32S3 端需要实现

```
1. HTTP下载固件(.bin或.fot文件)
2. 按 UART_S_UPDATE 协议打包发送
   - 握手: "START_UPD^_^"
   - 命令: 0x55AA + cmd + status + addr + len + crc
   - 数据: 512字节/包
   - 查询: 读取烧写状态
3. 完成后 AB5605B WDT 自动重启
```

### 关键风险提醒

```
UART 有线方案虽然断连风险极低，但仍有不到 1% 的掉电变砖概率。
如果产品不能接受任何变砖风险，唯一的方案是：
  → 双 Flash / 外部 Flash  + 切换机制（但 AB5605B 不支持双 Bank）
```
