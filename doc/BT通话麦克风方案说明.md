# 蓝牙通话麦克风方案说明



![](..\pic\1111.svg)



#### 为什么市面上没见过这种产品

**A2DP** 只管音乐。要做**通话**，蓝牙音响必须自己带 **MIC，必须走 HFP**。市面上所有的通话蓝牙产品（车载蓝牙、小爱音箱、AirPods、蓝牙耳机）全都有自己的 MIC，没有例外。

#### 正常耳机模式/蓝牙音响（音响出声 + 音响自己的 MIC）

```
                        手机
                        │
                        ├── HFP 下行 ──▶ 蓝牙芯片 ──▶ DAC ──▶ 喇叭
                        │                                   ✅ 对方说话 → 音响出声音
                        │
                        └── HFP 上行 ◀── 蓝牙芯片 ◀── MIC 采集
                                                           ✅ 板子MIC → 对方听到
```

- ✅ 对方声音：手机 → 音响（HFP 下行）
- ✅ 我说话：MIC → 手机（HFP 上行）

## 一、核心概念：蓝牙的两个音频通道

蓝牙音响和手机之间有两**条独立的音频通道**，分别对应两个协议：

```
┌────────────────────────────────────────────────────────────┐
│                                                             │
│   🎵 A2DP（音乐通道）       📞 HFP（通话通道）                  │
│                                                             │
│   手机 ──音乐──▶ 音响      手机 ──对方声音──▶ 音响               │
│                            音响 ──我说话──────▶ 手机          │
│                                                             │
│   只下传，单向             上下传双向，对讲模式                   │
│   带宽大，音质好            带宽小，音质一般                      │
│                                                             │
└────────────────────────────────────────────────────────────┘

关键区别：A2DP 只管"放音乐"，HFP 要同时"听对方"和"传给对方"。
```

```
放音乐时：  A2DP ────────────▶ 音响 ✅
接电话时：  A2DP 被暂停 ──────
          HFP ────────────▶ 音响（对面声音）✅
          HFP ◀──────────── 音响 MIC（你说话）❌ 板子没 MIC
```

**电话一来，音乐通道就关掉了，只剩通话通道在工作。**

## 二、为什么做不到

![](..\pic\2222.svg)



### 原因1：HFP 一建立，手机自动关闭自己的 MIC

```
没有 HFP 连接时：
  手机 MIC 开着 ── 跟普通打电话一样
  
HFP 连接建立后：
  手机认为"话筒在蓝牙设备上" ── 把自己的 MIC 关了
  所有语音采集责任转给蓝牙设备
```

这是手机操作系统的标准行为（iOS/Android 都一样），不是我们代码能控制的。

### 原因2：HFP 上行和下行绑定在同一条 SCO 链路

HFP 底层的 SCO（同步面向连接）链路**本身就是双向的**：

```
┌────────────── 蓝牙 HFP (SCO链路) ──────────────┐
│                                                  │
│  下行：对方的声音 → 喇叭                             │
│  上行：MIC 采集   → 对方                            │
│                                                  │
│    上行和下行在物理上是同一条链路，拆不开               │
│                                                  │
└──────────────────────────────────────────────────┘
```

如果要"上行走手机 MIC、下行走音箱喇叭"，意味着：

```
把这根线：  ─────────────
砍成两半：  ─────    ─────
一半接手机MIC，一半接音箱    ← 蓝牙协议不支持这种操作
```

## 三、代码执行证明链路拆不开

下面从本 SDK 代码的执行流程，直观展示 HFP 通话时音频是怎么"绑"在一起的。

#### 4.2 为什么不能在代码里"拆开"上下行

```
HFP 通话时，SCO 编解码器的工作模型（蓝牙库内部，代码不可改）：

   ┌─────────────── bt_voice 编解码器 ───────────────┐
   │                                                   │
   │  SCO 下行 ◀── 蓝牙射频 ◀── 手机 ◀── 对方说话        │
   │     │                                             │
   │     └──▶ bt_voice_decode ──▶ DAC ──▶ 喇叭         │
   │                                                   │
   │  喇叭 MIC ◀── SDADC ◀── btmic_ch_tbl[x]           │
   │     │                                             │
   │     └──▶ bt_voice_encode ──▶ 蓝牙射频 ──▶ 手机     │
   │                                                   │
   │  上行和下行在同一个 bt_voice 实例里处理              │
   │  蓝牙库闭源，无法修改                               │
   └───────────────────────────────────────────────────┘

关键的代码位置：
  btmic_ch_tbl[] = { 0, CH_MIC_PF2, CH_MIC_PF3 }  ← MIC 只能是物理引脚
  xcfg_cb.bt_ch_mic                               ← 配置工具选 MIC 通道
  bt_voice_cfg.mic_eq_en                           ← MIC EQ 开关

  没有任何一行代码可以"把手机 MIC 的数据灌进 SCO 上行"
```

#### 4.3 换一个芯片、用RTOS 多线程调度，能不能实现？

**答案是：不能。原因跟芯片、操作系统都无关，这是蓝牙协议层的硬限制。**

```
  ┌─────────────────────────────────────────────────────────────┐
  │                  蓝牙协议栈分层                               │
  │                                                             │
  │  应用层（你的代码）                                         │
  │    ├─ 单循环 or RTOS 多线程    ← 改这层没用                   │
  │    │                                                         │
  │  Profile 层（HFP/A2DP）                                      │
  │    ├─ 定义通话行为规范                                        │
  │    │                                                         │
  │  Host 层（蓝牙协议栈）                                       │
  │    ├─ SCO 链路管理                                          │
  │    │                                                         │
  │  Controller 层（基带/射频硬件）  ← 问题在这里，芯片硬件决定的  │
  │    └─ SCO 时隙是双向的，一个时隙同时收发语音                  │
  │       上行从 MIC 引脚 → 射频发出                              │
  │       下行从 射频收到 → DAC 播放                              │
  │       这两条路径在硅片上就绑定了，上层软件没法拆               │
  └─────────────────────────────────────────────────────────────┘

  换一个 SoC 也一样（高通、瑞昱、杰理……）：
  蓝牙 HFP 的 SCO 链路规范是 SIG 标准，所有芯片都遵循同一份协议。
  不存在"某个芯片的 SDK 可以拆开上下行"的可能性。

  RTOS 的任务调度能解决"CPU 时间分配"问题，不能解决"射频物理链路方向"问题。

```
