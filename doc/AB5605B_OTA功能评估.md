# AB5605B OTA（FOTA）功能评估

## 一、SDK 已有能力

SDK 已内置 FOTA 升级框架，代码路径：

```
app_fota.h/c     → platform/modules/app/app_fota/    (协议层)
api_update.h     → platform/libs/                    (底层 flash 写操作)
sfunc_bt_ota.c   → platform/functions/               (传统 OTA 入口)
```

### 支持能力

| 特性 | 状态 |
|------|------|
| 传输通道 | ✅ SPP（BT 经典）+ BLE GATT Notify |
| 通信加密 | ✅ 7字节接入码认证 |
| 断点续传 | ✅ 支持（记录已烧写地址到 flash 参数区） |
| 差分/压缩 | ❌ 不支持，全量传输 |
| 双备份 | ❌ 不支持，直接覆写运行区 |

### 使能方式

```c
// config.h 中打开以下任一：
#define BT_AB_FOT_EN 1   // SPP FOTA（需同时开 BT_SPP_EN）
#define LE_AB_FOT_EN 1  // BLE FOTA（需同时开 LE_EN）

// config_extra.h 自动推导出：
#define FOT_EN 1         // FOTA 总开关
```

### 传输流程

```
手机 APP  →  建立 SPP/BLE 连接
          →  发送 7 字节接入码认证
          →  获取设备版本信息
          →  发起升级请求（含固件大小、哈希值）
          →  分片传输（每包 ~140 字节 / 512 块对齐）
          →  设备写入 flash
          →  全部完成后校验
          →  延时 3 秒，WDT 复位生效
```

## 二、风险评估（你提到的变砖问题）

### 风险点①：单 Bank 直接覆写

```
Flash 布局（512KB）：

 ┌─────────────────┐  0x00000
 │   Init + 代码区   │  492KB ← 运行中的程序就存在这里
 │   (492KB)        │          OTA 时直接往这块写入新固件
 ├─────────────────┤  0x7B000
 │   配置文件/参数    │
 ├─────────────────┤  0x7E000
 │   FOTA 缓冲区     │
 └─────────────────┘  0x80000 (芯片内置 flash 最大 512KB)

升级时：边接收边擦除边写入 → 任意中断 → 代码不完整 → 变砖
```

### 风险点②：擦除周期长

- Flash 块大小 512 字节，擦除以 4K 为单位
- 492KB / 4K = 123 次擦除
- 每次擦除 ~10ms → 总擦除时间 > 1 秒
- 每次写入 ~3ms → 总烧写时间 ~3 秒
- 这 4+ 秒里只要断电/断连任何一次就废了

### 风险点③：无恢复机制

| 场景 | 结果 |
|------|------|
| 升级中手机断连 | ❌ 变砖 |
| 升级中电池耗尽 | ❌ 变砖 |
| 升级中 flash 擦写错误 | ❌ 变砖 |
| 校验失败 | ❌ 变砖 |
| 复位后程序跑不起来 | ❌ 变砖 |

**没有"升级失败回滚到上一版"这种机制。** 想恢复只能拆机用 ISP/UART 重新烧录。

## 三、对比其他芯片的 OTA 方案

| 特性 | AB5605B | 常见做法（如 ESP32） |
|------|---------|-------------------|
| 双 Bank | ❌ 无 | ✅ 有，A/B 区交替 |
| 差分升级 | ❌ 无 | ✅ 部分支持 |
| 断点续传 | ✅ 有 | ✅ 有 |
| 变砖风险 | **高** | 低（可回滚） |
| 恢复方式 | 只能用 ISP 烧录器 | OTA 回滚 + 串口 |

## 四、建议

### 预研/原型阶段

**不建议启用 FOTA**。当前用 ISP 烧录器（USB 转 SPI/UART 工具）更新固件，稳定可靠。

```c
// config.h 保持关闭
#define BT_AB_FOT_EN 0
#define LE_AB_FOT_EN 0
```

### 量产阶段如果仍需 OTA

1. **硬件上增加保证**：必须保证传输期间不掉电、不断连。
2. **软件上缩短窗口**：当前 492KB 传输量大，考虑裁剪不必要的代码减少固件体积。
3. **按 block 校验**：每写完一个 block 做 CRC 确认，失败立即重传，尽量降低中段概率。
4. **用户提示**：OTA 过程中强提醒用户不要操作手机、保持信号稳定。

### 如果一定要用 OTA

```c
// 需要开启的配置
#define BT_AB_FOT_EN 1       // SPP 通道 OTA（比 BLE 更稳定）
#define BT_SPP_EN 1          // SPP 配套
// BLE FOTA 不推荐，BLE 连接不稳定，传输更慢
```

> **建议在产品定型前评估是否需要 OTA。** 如果目标客户的升级场景是"出厂前统一烧录"或"很少更新"，不启用 OTA 是最安全的选择。
